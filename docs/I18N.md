### Translation Guidelines

1. **Key Naming Convention**
   - Format: `<module>.<feature>.<message>`
   - Example: `setup.api_key.warning`, `cli.error.invalid_url`

2. **Format Strings**
   - Use Python standard format: `{variable_name}`
   - Example: `"Current API key: {key}..."`

3. **Consistency**
   - Use consistent terminology for same concepts
   - Example: "API í‚¤" (Korean), "API key" (English), "API å¯†é’¥" (Chinese)

4. **Cultural Adaptation**
   - Translate naturally for each language's culture
   - Emojis remain the same across all languages

### Contribution Guide

To contribute new languages or improve translations:

1. Create or edit `ytt/locales/<lang>.json`
2. Provide translations for all keys
3. Submit PR with:
   - Language code (ISO 639-1)
   - Translator information
   - Native speaker review status

## Implementation Details

### Auto-initialization

The i18n module automatically initializes on import by reading the user's language preference from the config file.

### Fallback Behavior

1. If a translation key is missing, returns `[key]`
2. If a language file is missing, falls back to Korean
3. If format variable is missing, prints warning and returns unformatted text

### Performance

- Translations are cached in memory after first load
- JSON files are only read once per language
- Minimal overhead on CLI startup

### Testing

```bash
# Test i18n module
python -c "from ytt.i18n import t, set_language; set_language('en'); print(t('setup.welcome.title'))"

# Run setup in different languages
ytt-init  # Will prompt for language
```
